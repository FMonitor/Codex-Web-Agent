# 项目开发需求设计

## 项目名称

**Copilot CLI Web Console**

## 补充说明（2026-04-18）

本期在原有范围基础上，临时新增以下要求：

1. 增加对 **Codex CLI** 的支持
2. 后端 adapter 设计需同时适配：
   - Copilot CLI bridge 风格的长连接/JSON Lines 协议
   - Codex CLI 官方 `codex exec --json` / `codex exec resume ... --json` 协议
3. 提供一个可运行的 **容器内 Codex CLI 环境**
4. 容器内 Codex CLI 环境需支持两类认证/服务模式：
   - OpenAI / ChatGPT 登录
   - 自定义 API（OpenAI-compatible provider）
5. 若通过代理访问 OpenAI 模型，需预留 `requires_openai_auth = true` 这种官方配置方式

## 一、项目目标

构建一个基于 Web 的轻量控制台，用于连接**单个 Copilot CLI 会话**，并在浏览器中提供接近原生 Agent 的交互体验。

系统首期目标：

1. 用户可通过 **WebUI** 与本机运行的 **Copilot CLI** 进行对话
2. WebUI 可实时展示 Copilot CLI 的执行状态与流式输出
3. WebUI 可展示工具调用、执行进度、文件变更、日志等原生能力
4. 支持手机浏览器访问
5. 架构上预留未来扩展到**多 Agent 协作**的能力

本期**不要求**实现：

* Todo 面板
* Default / Bypass / Autopilot 模式切换
* 多 Agent 协同执行
* 复杂审批策略界面
* 完整 IDE 替代体验

---

## 二、项目定位

本项目不是一个完整 IDE，也不是一个重新实现 Copilot 的系统。

本项目的定位是：

> 一个面向手机和浏览器的 Copilot CLI 会话控制台
> 用于承接单个 Agent 的对话、执行状态展示和基础操作控制

---

## 三、产品形态

### 3.1 用户使用流程

用户打开 WebUI 后，可执行以下流程：

1. 创建或连接一个 Copilot CLI 会话
2. 输入任务，例如：

   * 修复某个模块
   * 分析某段代码
   * 生成某个文件
   * 运行测试并修复失败项
3. WebUI 实时显示：

   * 模型回复
   * 当前执行状态
   * 工具调用信息
   * 文件变更摘要
   * 日志输出
4. 用户可在执行过程中继续发送补充指令
5. 用户可停止当前执行，或查看本轮执行结果

## 五、总体架构

系统由三层组成：

### 5.1 WebUI

负责：

* 聊天界面
* 状态展示
* 工具调用展示
* 文件变更展示
* 日志输出展示
* 会话控制按钮
* 移动端适配

### 5.2 Backend Gateway

负责：

* 对接 Copilot CLI
* 维护 Web 端 session
* 转发用户消息
* 接收并整理 CLI 事件流
* 向前端推送 SSE / WebSocket 事件
* 做统一的数据结构转换

### 5.3 Copilot CLI Runtime

负责：

* 真正执行 Agent 会话
* 访问代码仓库
* 读取/修改文件
* 调用 shell / tools
* 返回原始执行事件

## 六、核心设计原则

### 6.1 尽量复用 Copilot CLI 原生能力

不要重新发明 agent loop、工具调用和执行逻辑。
WebUI 的主要职责是：

* 展示
* 控制
* 转换
* 扩展

而不是替代 Copilot CLI。

### 6.2 前后端都围绕“事件流”设计

系统核心不是请求-响应，而是**持续事件流**。
所有 UI 状态都应由事件驱动。

### 6.3 先做单 Agent，会话模型保持可扩展

虽然本期只实现一个 Agent，但底层 session / adapter / runtime 设计必须支持未来扩展为：

* 多个 Copilot CLI worker
* 多角色 agent
* 协调器编排

## 七、功能需求

# 7.1 会话管理

## 目标

系统支持创建和管理一个活跃的 Copilot CLI 会话。

## 要求

* 可创建新会话
* 可关闭当前会话
* 可查询当前会话状态
* 可恢复最近一次会话（可选）
* 同一时间 MVP 阶段仅允许一个活跃执行会话

## Session 数据结构建议

```json
{
  "id": "sess_001",
  "title": "修复登录模块",
  "status": "idle",
  "workspacePath": "/project/demo",
  "createdAt": "2026-04-18T10:00:00Z",
  "updatedAt": "2026-04-18T10:00:00Z",
  "runtime": "copilot-cli",
  "agentId": "default"
}
```

## Session 状态

* `idle`
* `running`
* `waiting_input`
* `stopped`
* `completed`
* `failed`

---

# 7.2 聊天输入

## 目标

用户可以像聊天一样提交任务给 Copilot CLI。

## 要求

* 支持发送文本消息
* 支持在执行过程中发送补充消息
* 支持停止当前执行
* 支持显示最后一条用户消息和最后一条 agent 回复

## 消息结构建议

```json
{
  "id": "msg_001",
  "sessionId": "sess_001",
  "role": "user",
  "content": "请分析 auth 模块，并修复登录逻辑",
  "createdAt": "2026-04-18T10:01:00Z"
}
```

---

# 7.3 执行状态展示

## 目标

WebUI 必须能显示 Copilot CLI 的“工作过程”，而不只是最终答案。

## 展示内容

前端至少支持以下状态类别：

* Thinking
* Planning
* Searching
* Reading
* Editing
* Creating File
* Running Command
* Waiting Approval
* Summarizing
* Completed
* Failed

## 实现方式

后端接收 Copilot CLI 原始事件后，转换为统一事件模型，再推送给前端。

## 统一事件结构建议

```json
{
  "id": "evt_001",
  "sessionId": "sess_001",
  "type": "tool.execution_start",
  "phase": "searching",
  "title": "搜索认证模块相关代码",
  "message": "正在查找 auth/login 相关实现",
  "timestamp": "2026-04-18T10:01:10Z",
  "raw": {}
}
```

---

# 7.4 工具调用展示

## 目标

用户可以看到 Agent 调用了什么工具，以及这些工具的执行状态。

## 要求

前端展示：

* 工具名称
* 开始时间
* 执行状态
* 简短参数摘要
* 执行结果摘要
* 失败原因（若有）

## 工具执行状态

* `pending`
* `running`
* `completed`
* `failed`
* `cancelled`

## 工具调用结构建议

```json
{
  "id": "tool_001",
  "sessionId": "sess_001",
  "name": "read_file",
  "status": "running",
  "summary": "读取 src/auth/login.ts",
  "startedAt": "2026-04-18T10:01:12Z",
  "endedAt": null
}
```

---

# 7.5 文件变更展示

## 目标

展示 Agent 本轮执行中涉及的文件变更。

## 要求

至少支持展示：

* 文件路径
* 变更类型：新增 / 修改 / 删除
* 变更摘要
* 是否为本轮新变更

## 数据结构建议

```json
{
  "id": "chg_001",
  "sessionId": "sess_001",
  "path": "src/auth/login.ts",
  "changeType": "modified",
  "summary": "增加 token 校验逻辑",
  "timestamp": "2026-04-18T10:01:40Z"
}
```

## 本期可不做

* 完整 diff viewer
* 行级 patch 可视化

但接口上要预留 `patch` 字段。

# 7.6 日志输出展示

## 八、接口设计

# 8.1 REST API

## 创建会话

`POST /api/sessions`

请求：

```json
{
  "title": "修复认证逻辑",
  "workspacePath": "/project/demo"
}
```

返回：

```json
{
  "sessionId": "sess_001"
}
```

---

## 发送消息

`POST /api/sessions/:sessionId/messages`

请求：

```json
{
  "content": "请检查登录逻辑并修复异常处理"
}
```

返回：

```json
{
  "accepted": true
}
```

---

## 停止执行

`POST /api/sessions/:sessionId/stop`

返回：

```json
{
  "accepted": true
}
```

---

## 获取会话状态

`GET /api/sessions/:sessionId`

返回：

```json
{
  "id": "sess_001",
  "status": "running"
}
```

---

# 8.2 事件流接口

推荐使用 **SSE**，实现简单，适合移动端。

## 订阅事件流

`GET /api/sessions/:sessionId/events`

服务端持续推送事件：

```json
{
  "type": "assistant.message_delta",
  "sessionId": "sess_001",
  "content": "我正在检查登录流程..."
}
```

## 十、后端设计

# 10.1 Backend Gateway 责任

* 启动或连接 Copilot CLI 会话
* 将 CLI 原始事件转换成统一格式
* 向 WebUI 推送实时事件
* 管理 session 生命周期
* 记录最基础的会话状态

# 10.2 Runtime Adapter 设计

为了预留多 Agent 扩展，Copilot CLI 的接入必须通过统一适配器接口完成。

## RuntimeAdapter 接口建议

```ts
interface RuntimeAdapter {
  createSession(input: CreateSessionInput): Promise<Session>;
  sendMessage(sessionId: string, content: string): Promise<void>;
  stopSession(sessionId: string): Promise<void>;
  subscribe(sessionId: string, onEvent: (event: RuntimeEvent) => void): Promise<void>;
  disposeSession(sessionId: string): Promise<void>;
}
```

## 当前实现

* `CopilotCliAdapter`

## 未来扩展

* `CopilotCliPmAdapter`
* `CopilotCliQaAdapter`
* `LocalModelAdapter`
* `CoordinatorAdapter`
